## 项目经历

### Ran Feed 内容社区后端系统

项目角色：后端核心开发  
技术栈：Go、GORM、MySQL、Redis、Kafka、Canal、Lua、JWT

#### 项目介绍

Ran Feed 是一个面向内容社区场景的单体后端系统，支持用户注册登录、用户主页、图文/视频内容发布、内容详情、推荐流、关注流、点赞、收藏、评论、关注、计数统计与文件上传等核心能力。项目采用清晰的业务分层架构，将内容、Feed、互动、计数、用户、上传、消息队列、缓存等模块解耦组织，保留了可演进为微服务的模块边界。

系统围绕内容流场景设计了多条数据链路：推荐流基于 Redis 热榜快照读取，关注流基于用户 inbox 缓存和冷启动回源重建，点赞等高频互动优先写入 Redis 并通过 Kafka 异步落库，计数系统通过 Canal/Kafka 订阅 MySQL 变更并聚合到统一计数表，同时驱动缓存失效与热度增量更新。

#### 主要职责

- 设计并实现用户、内容、互动、Feed、计数、上传等核心模块，提供注册登录、内容发布、内容详情、推荐流、关注流、点赞收藏评论关注等 REST API。
- 设计 MySQL 核心表结构，包括用户表、内容主表、文章/视频详情表、点赞/收藏/评论/关注关系表、统一计数表与消息消费幂等表，并针对内容流、作者页、热榜、互动关系建立索引。
- 实现 Redis 缓存体系，使用 ZSet 承载推荐热榜、关注流 inbox、用户发布列表和收藏列表，使用 Hash 缓存用户点赞状态和热度增量，使用分布式锁控制缓存重建与定时任务并发。
- 实现点赞异步链路，用户点赞/取消点赞先通过 Redis Lua 脚本完成状态切换，再投递 Kafka 事件，由消费者幂等落库并同步计数与热度变化。
- 实现 Canal + Kafka 计数聚合链路，订阅点赞、收藏、评论、关注、内容变更事件，按表分发处理策略，更新统一计数表并失效内容计数、用户主页计数等缓存。
- 实现热榜更新机制，将发布、点赞、收藏、评论等行为转化为热度增量写入 Redis 分片 Hash，通过定时任务合并增量到全局 ZSet，构建推荐流快照并同步 MySQL 热度字段。
- 实现关注流能力，内容发布后异步 fanout 到粉丝 inbox；当用户 inbox 缓存不存在时，基于关注关系列表和作者近期内容回源重建，提升关注流读取效率和可用性。
- 实现内容详情与 Feed 列表的并行信息补全，通过并发任务编排查询作者信息、点赞状态、收藏状态、关注状态和计数信息，降低接口聚合延迟。
- 实现 JWT 登录态和 Redis 双 Key 会话管理，支持登录、退出、鉴权、会话校验与续期。
- 实现本地对象上传与临时上传凭证，支持头像、文章封面、视频封面、视频源文件等场景的文件类型校验、大小限制、HMAC 签名和过期校验。

#### 技术亮点

- 高频互动异步化：针对点赞等高频写场景，将用户互动状态前置到 Redis Hash，并通过 Lua 脚本保证状态切换、TTL 刷新和容量控制的原子性；状态变化后投递 Kafka 事件异步落库，配合消费幂等表去重，降低 MySQL 写入压力并提升互动接口响应速度。
- Feed 流与热榜体系：基于 Redis ZSet + Cursor 分页实现推荐流、关注流、作者发布流和用户收藏流；内容发布时异步 fanout 到粉丝 inbox，缓存缺失时通过分布式锁回源重建；推荐流通过热度增量分片 Hash、定时合并、快照读取的方式提升榜单读取稳定性。
- 事件驱动计数一致性：通过 Canal 订阅 MySQL Binlog，将点赞、收藏、评论、关注、内容删除等变更统一转化为计数变更事件，更新统一计数真相表，并主动失效内容计数、用户主页计数等缓存，保障高频互动下的最终一致性。
- 聚合接口性能优化：内容详情和 Feed 列表通过批量查询与并发聚合补全作者、点赞状态、收藏状态、关注状态和计数信息，减少串行等待和 N+1 查询，降低内容流和详情页接口的聚合延迟。
